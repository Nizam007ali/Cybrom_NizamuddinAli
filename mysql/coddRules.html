Codd rules

E.F. Codd, an English computer scientist, proposed 12 rules (often called Codd's 12 Rules) to define what is required from a database management system (DBMS) to be considered truly relational. Although he proposed these rules in the 1980s, they are still foundational in understanding relational database management systems (RDBMS).
Here are the 12 rules:

  1. Information Rule: All information (including metadata) is represented logically in tables.
  2. Guaranteed Access Rule: Every piece of data is accessible through a combination of table name, primary key (row), and attribute (column).
  3. Systematic Treatment of NULL Values: NULLs (unknown or missing data) must be uniformly treated as distinct values, independent of data type.
  4. Dynamic Online Catalog Based on the Relational Model: The database metadata should be stored in tables and be accessible using the same query language used to access regular data.
  5. Comprehensive Data Sublanguage Rule: A relational system may support several languages, but there must be at least one language that supports data definition, manipulation, integrity constraints, and transaction management.
  6. View Updating Rule: Any view that is theoretically updatable must be updatable through the system.
  7. High-level Insert, Update, and Delete: The system must support insertions, updates, and deletions at a set level (not just a single row).
  8. Physical Data Independence: Changes to the physical storage of data should not affect how data is accessed by users.
  9. Logical Data Independence: Changes to the logical structure (tables) of the database should not affect the user’s ability to access data.
  10. Integrity Independence: Integrity constraints must be defined separately from application programs and stored in the catalog.
  11. Distribution Independence: The end-user should not be aware of whether the database is distributed across multiple locations.
  12. Non-subversion Rule: If a system provides low-level access to the data, it should not allow it to bypass integrity rules defined at a higher level.
These rules were intended to be a guideline for developing fully relational databases, although no database system fully adheres to all of them even today.

The breakdown of which Codd's rules are commonly not fully followed by major RDBMSs:

1. Oracle Database:


Rule 6 (View Updating Rule): Complex views, particularly those involving joins, aggregations, or derived fields, may not always be directly updatable.
Rule 9 (Logical Data Independence): Structural changes in the database (like adding columns or modifying tables) can still impact applications.
Rule 11 (Distribution Independence): In distributed systems, Oracle does not fully abstract distribution from the end user.

2. Microsoft SQL Server:

Rule 6 (View Updating Rule): Complex views are not always updatable.
Rule 9 (Logical Data Independence): Changes to the logical schema (like splitting or merging tables) can affect applications.
Rule 11 (Distribution Independence): Distributed databases and sharding are not entirely transparent.

3. MySQL:

Rule 6 (View Updating Rule): Complex views often cannot be updated directly.
Rule 9 (Logical Data Independence): Applications can still be impacted by schema changes like adding or altering tables.
Rule 11 (Distribution Independence): Full distribution transparency is not achieved in MySQL’s distributed setups.

4. PostgreSQL:

Rule 6 (View Updating Rule): PostgreSQL supports view updates, but not for highly complex or derived views.
Rule 11 (Distribution Independence): PostgreSQL does not completely hide the distributed nature of data across multiple servers.

5. IBM Db2:

Rule 6 (View Updating Rule): Db2 struggles with updating views that involve complex logic or aggregations.
Rule 9 (Logical Data Independence): Changes in the schema can still require application-level adjustments.
